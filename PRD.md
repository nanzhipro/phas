# 基于 Apple 原生虚拟化的本地 Linux VM MVP PRD

## 1. 项目目标

基于 Apple 官方 Virtualization.framework，在 macOS 本地创建并运行一个 Linux 虚拟机，先做出一个可交付的 MVP 版本。

这个 MVP 的目标不是做成完整的 Parallels/VMware 替代品，而是做成一个范围克制、但从用户视角“基本可用且闭环完整”的本地 Linux 虚拟机产品。

MVP 必须满足以下结果：

- 用户可以在 macOS 上创建 1 台 Linux VM。
- 用户可以通过图形化安装流程完成 Linux 安装。
- 用户可以在安装完成后重复启动、使用、关闭这台 VM。
- VM 具备基础网络能力，可以访问互联网。
- VM 的系统盘和配置可以持久保存，不是一次性 demo。

## 2. 调研结论与产品选型

基于 Apple 官方文档调研，得到以下结论：

1. Apple 官方 Virtualization.framework 支持在 Apple silicon 和 Intel Mac 上运行 Linux 虚拟机，但 guest 的 CPU 架构必须与 host 匹配。
2. Apple 官方给出了两条 Linux 路径：
   - 最小技术演示路径：直接加载 Linux kernel + initrd，通过串口控制台进入系统。
   - 可交付产品路径：通过 ISO 安装镜像 + EFI + 图形设备创建一个可持久运行的 Linux VM。
3. 如果目标是“验证框架可用”，第一条路径足够；如果目标是“MVP 精简但基本功能完整”，第二条路径更合适。
4. NAT 网络是 Apple 原生支持且默认成本最低的联网方式，不需要额外的 `com.apple.vm.networking` entitlement；bridged networking 需要额外 entitlement，不适合放入首版。
5. 共享目录和剪贴板共享虽然 Apple 原生支持，但都依赖 guest 侧能力：
   - 共享目录依赖 virtiofs。
   - 剪贴板共享依赖 `spice-vdagent`。
   因此这两项不应进入 MVP 的必做范围。

### 2.1 MVP 方案结论

MVP 采用以下产品形态：

- 原生 macOS 应用。
- 基于 Apple Virtualization.framework 实现。
- 使用 Linux ISO 安装镜像创建虚拟机。
- 使用 EFI 启动与持久化虚拟磁盘。
- 提供图形界面与键鼠交互。
- 默认 NAT 网络。
- 首版仅支持单虚拟机闭环，不做多 VM 管理平台。

### 2.2 为什么不选“kernel + initrd + 串口”作为 MVP

该方案虽然最轻，但它更像框架样例，而不是可交付产品：

- 用户需要自行准备 kernel 和 initrd，门槛高。
- 安装路径不直观，不适合作为普通用户工作流。
- 更适合开发验证，不适合作为首版产品体验。

因此，本 PRD 将 ISO 安装型 GUI Linux VM 定义为 MVP 主路径。

## 3. 产品定义

这是一个运行在 macOS 本地的原生 Linux 虚拟机工具，核心职责是：

- 创建一台 Linux VM。
- 完成首次安装。
- 保存 VM 状态与磁盘。
- 后续重复启动与使用。
- 提供基础网络访问能力。

产品不承担以下职责：

- 企业级虚拟机编排。
- 多镜像仓库管理。
- 云端同步。
- 跨架构仿真。
- 完整桌面虚拟化平台替代。

### 3.1 产品原则

为避免 MVP 在实现阶段失控，产品必须遵守以下原则：

1. 单虚拟机闭环优先于多功能堆叠。首版只要把一台 VM 的创建、安装、启动、关闭、删除做完整，就算成功。
2. 默认值优先于开放配置。除 CPU、内存、磁盘外，不向用户暴露大量底层虚拟化参数。
3. 可恢复优先于自动化魔法。发生错误时，优先保留 VM 数据并给出可重试路径，而不是静默修改 VM 状态。
4. 宿主机安全优先于 guest 性能。资源分配、删除操作、文件访问都要优先避免影响宿主机稳定性。
5. guest 自治边界清晰。产品负责创建和运行 Linux VM，但不负责接管 Linux 发行版安装器或 guest 内部配置。

## 4. 目标用户与核心场景

### 4.1 目标用户

- 需要在 Mac 上运行 Linux 环境的开发者。
- 需要隔离测试环境的工程师。
- 需要一个本地、干净、可重复启动的 Linux 工作空间的用户。

### 4.2 核心场景

1. 作为开发者，我希望在 Mac 上快速创建一台 Linux VM，用于运行 Linux-only 工具或脚本。
2. 作为测试用户，我希望 VM 具备图形界面和网络能力，能像普通 Linux 机器一样完成安装与日常使用。
3. 作为普通使用者，我希望关闭应用后，下次还能继续启动之前安装好的 Linux 系统，而不是每次从零开始。

## 5. MVP 范围

### 5.1 必做功能

#### A. 创建虚拟机

- 提供“新建 VM”流程。
- 用户可以输入 VM 名称。
- 用户可以选择 Linux ISO 安装镜像。
- 用户可以配置 CPU、内存、磁盘大小。
- 系统会创建该 VM 对应的持久化目录与虚拟磁盘。

#### B. 首次安装流程

- 首次启动时，从 ISO 镜像引导进入 Linux 安装器。
- 用户在 VM 图形窗口中完成 Linux 安装。
- 安装完成并关机后，后续启动默认从虚拟磁盘引导，不再重复进入安装流程。

#### C. 运行与交互

- 提供 VM 图形显示窗口。
- 支持键盘和鼠标输入。
- 支持窗口关闭、重新打开、再次进入 VM。
- 支持基础状态展示：未安装、已安装、运行中、已停止、错误。

#### D. 生命周期管理

- 启动 VM。
- 关闭 VM。
- 检测 guest 主动关机。
- 强制停止 VM。
- 删除 VM 及其本地数据。

#### E. 网络能力

- 默认使用 NAT 网络。
- guest 在安装完成后可以访问互联网。
- 不做自定义网段、不做端口转发 UI、不做 bridged networking。

#### F. 基础错误处理

- 对常见错误提供明确提示：
  - ISO 文件不存在或不可读。
  - 主机架构与镜像架构不匹配。
  - 磁盘空间不足。
  - 虚拟机配置校验失败。
  - 启动失败。

### 5.2 明确不做

以下能力不进入 MVP：

- 多台 VM 同时管理。
- VM 快照、克隆、回滚。
- Bridged networking。
- 自定义网络拓扑。
- 共享目录。
- 宿主机与 guest 的剪贴板共享。
- USB 设备透传。
- 音频输入输出。
- Rosetta for Linux。
- 导入现有 qcow2、vmdk 等第三方磁盘格式。
- 自动下载 Linux 镜像。
- 无人值守安装。

## 6. MVP 产品形态

### 6.1 交互形态

MVP 采用原生 macOS GUI 应用。

原因：

- Apple 官方已有 GUI Linux VM 的完整样例路径。
- 从用户体验上，GUI 安装路径比串口方案更符合“基本功能完整”。
- 生命周期和安装状态管理也更容易被普通用户理解。

### 6.2 首版支持矩阵

为控制范围，MVP 首版锁定以下支持矩阵：

- 宿主机：Apple silicon Mac。
- 宿主系统：macOS 14 及以上。
- 主验收发行版：Ubuntu Desktop ARM64（当前受支持 LTS）。
- 补充验证发行版：Fedora Workstation ARM64。

说明：

- Apple 官方框架也支持 Intel Mac，但 Intel 支持会增加镜像矩阵、测试矩阵和适配复杂度。
- 因此 Intel 支持保留为后续阶段，而不是 MVP 验收条件。
- 首版对外承诺以 Ubuntu Desktop ARM64 为准，Fedora Workstation ARM64 仅作为补充兼容性验证，不作为发布阻塞条件。
- Apple 官方 GUI Linux 示例路径基于 macOS 13.0+；本 PRD 选择 macOS 14+ 是产品收敛决策，不是官方 API 的最低硬性要求。

### 6.3 MVP 信息架构

MVP 建议只保留最少的 GUI 结构：

- 首页或空状态页：展示“尚未创建 VM”或唯一一台 VM 的摘要。
- 创建向导：负责名称、ISO、资源配置。
- VM 详情页：展示状态、配置摘要、启动/停止/删除入口。
- VM 运行窗口：承载 `VZVirtualMachineView`，只负责 guest 图形交互。
- 错误弹窗或错误区域：展示失败原因与重试动作。

首版不需要复杂的侧边栏、多标签页或多窗口管理系统。

## 7. 用户流程

### 7.1 新建 VM

1. 用户打开应用。
2. 点击“创建虚拟机”。
3. 输入名称。
4. 选择 Linux ISO。
5. 选择资源配置：CPU、内存、磁盘。
6. 点击创建。

### 7.2 首次安装

1. 系统创建 VM bundle 和虚拟磁盘。
2. 系统以 ISO 作为安装介质启动 VM。
3. 用户在图形安装器内完成 Linux 安装。
4. 安装完成后，guest 可能正常关机，或由安装器触发重启并从系统盘继续启动。
5. 应用在确认系统盘可引导后，把 VM 状态更新为“已安装”。

### 7.3 后续使用

1. 用户重新启动 VM。
2. VM 从虚拟磁盘启动。
3. 用户正常使用 Linux。
4. 用户可在 guest 内关机，或由宿主机触发强制停止。

## 8. 功能需求明细

### 8.1 VM 配置能力

MVP 只提供少量、清晰的配置项：

- VM 名称。
- ISO 路径。
- CPU 数量。
- 内存大小。
- 磁盘大小。

建议提供预设，而不是开放大量自由参数：

- 轻量：2 vCPU / 4 GiB / 32 GiB
- 标准：4 vCPU / 8 GiB / 64 GiB

如果宿主机资源不足，应用应提示并阻止创建。

创建向导应展示最小必要的宿主机信息，用于帮助用户选择配置：

- 宿主机总内存。
- 宿主机可用 CPU 核心数的产品级提示。
- 当前机器的推荐预设。
- 某个选项不可选时的原因说明。

MVP 不需要展示复杂的实时资源监控、曲线图或系统诊断面板。

### 8.1.1 资源分配约束

为避免用户创建出“理论可配、实际不可用”的 VM，MVP 需要加入资源约束：

- CPU 最低为 2 vCPU。
- 内存最低为 4 GiB，因为 MVP 的主路径是桌面版 Linux ISO 安装。
- 系统盘最低为 32 GiB。
- 默认不允许把宿主机全部资源分给 guest。
- 如果配置超过产品设定的安全阈值，应用应阻止创建并提示原因，而不是允许创建后在启动阶段失败。

具体阈值可以在技术设计中结合宿主机能力进一步细化，但 PRD 层面要求“有硬性保护，不做完全自由输入”。

### 8.1.2 创建前准入检查

在用户点击“创建”并真正落盘 VM 数据之前，产品必须执行一次创建前准入检查。

硬性拦截项：

- 宿主机不是 Apple silicon Mac。
- 宿主机 macOS 版本低于 MVP 要求。
- ISO 文件不存在、不可读或明显不是有效安装镜像。
- ISO 架构与宿主机架构不匹配。
- 可用磁盘空间不足以容纳 VM bundle 的创建和安装过程。
- CPU、内存、磁盘配置不满足产品下限或超过安全阈值。

软性警告项：

- 选择的镜像不是首版主验收发行版 Ubuntu Desktop ARM64。
- 选择的镜像不是补充验证发行版 Fedora Workstation ARM64。
- 镜像来源未知，产品无法承诺兼容性。

处理原则：

- 硬性拦截项必须阻止创建。
- 软性警告项可以允许继续，但必须明确提示“可尝试，非首版保证范围”。

### 8.1.3 配置可变更性

MVP 必须明确哪些配置在创建后还能改，哪些不能改：

- VM 唯一 ID 不可变。
- CPU、内存、磁盘大小在 MVP 中视为创建后不可编辑。
- 已安装 VM 不支持切换安装镜像。
- 仅 `Draft`、`Installing`、`Error` 状态允许重新选择 ISO。

如果用户想修改硬件规格，MVP 路径是删除并重建 VM，而不是提供复杂的重配置流程。

### 8.2 持久化策略

每台 VM 使用一个独立的本地 bundle 目录保存运行所需数据。

首版固定目录结构：

`~/Library/Application Support/phas/VMs/<vm-id>.vmbundle/`

建议内容：

- `Disk.img`：虚拟磁盘。
- `MachineIdentifier`：机器标识。
- `NVRAM`：EFI variable store。
- `config.json`：产品层 VM 元数据。
- `logs/`：启动与错误日志。

补充要求：

- `config.json` 必须包含 schema version，避免后续版本升级时无法迁移。
- 虚拟磁盘应优先采用对宿主机更友好的稀疏分配策略，而不是创建时一次性占满全部磁盘容量。
- 删除 VM 时，以 bundle 为最小清理单元，避免残留孤立文件。

### 8.3 状态模型

MVP 至少定义以下状态：

- Draft：刚创建，尚未成功进入安装流程。
- Installing：已进入安装流程，但尚未完成安装。
- Stopped：已安装，当前关机。
- Running：正在运行。
- Error：创建或启动失败，需要用户处理。

### 8.3.1 状态转移规则

MVP 必须对状态转移做出明确约束：

- Draft -> Installing：VM 元数据、磁盘与启动配置创建成功，并发起首次安装启动。
- Draft -> Error：创建阶段失败，例如 ISO 无效、磁盘创建失败、配置校验失败。
- Installing -> Stopped：安装流程完成，guest 正常关机，且后续应从系统盘启动。
- Installing -> Running：安装流程完成后，guest 通过安装器触发的重启或后续启动，已从系统盘进入可用系统。
- Installing -> Error：安装阶段启动失败或关键运行错误。
- Stopped -> Running：从系统盘成功启动。
- Running -> Stopped：guest 正常关机。
- Running -> Error：运行过程中发生无法恢复的错误。
- Error -> Installing 或 Stopped：用户修复问题后重试成功，回到对应稳定状态。

补充说明：

- `Error` 是处置状态，不等于必须销毁 VM 数据。
- 只要 bundle 和磁盘仍然可用，产品应优先支持“修复后重试”。

### 8.3.2 状态下的动作权限

为避免 UI 和状态机脱节，MVP 需要定义不同状态下允许的用户动作：

- `Draft`：允许编辑配置、重新选择 ISO、启动安装、删除。
- `Installing`：允许打开运行窗口、强制停止、重新选择 ISO、重试安装、删除。
- `Stopped`：允许启动、查看详情、查看日志、删除。
- `Running`：允许打开运行窗口、请求关机、强制停止、查看详情；不允许删除和修改配置。
- `Error`：允许查看错误信息、查看日志、重试、重新选择 ISO（如适用）、删除。

MVP 不应在某个状态下暴露“看起来能点、实际上不会成功”的动作。

### 8.4 删除策略

删除 VM 时应：

- 明确二次确认。
- 删除 bundle 下所有 VM 数据。
- 从应用列表中移除该 VM。

MVP 不做“回收站”和“快照备份”。

### 8.5 安装完成判定

MVP 不应依赖 guest 内部 agent，也不应要求读取 Linux 系统内部状态来判断安装是否完成。首版采用外部可观察原则：

1. 首次启动时使用 ISO 作为安装介质。
2. 用户在 guest 内完成安装；安装流程可能以正常关机收尾，也可能由安装器直接触发重启。
3. 后续启动或重启默认尝试从系统盘引导。
4. 如果系统盘可正常引导进入已安装系统，则 VM 视为安装完成。
5. 如果后续仍回到安装介质流程，或系统盘无法启动，则 VM 仍视为 `Installing` 或进入 `Error`，由用户重试。

这样可以避免为了判定安装完成而引入额外 guest 组件，保持 MVP 简洁。

### 8.6 失败场景与恢复策略

MVP 至少要覆盖以下失败场景：

- ISO 丢失或路径失效：提示重新选择 ISO，不直接删除 VM。
- 宿主机空间不足：阻止创建或启动，并提示需要的空间。
- 虚拟机配置校验失败：保留配置页面，让用户修正后重试。
- 首次安装中断：允许再次进入安装流程，不强制用户删除重建。
- 安装后首次从系统盘启动失败：标记为错误并提供重试或删除入口。
- 应用异常退出后重开：如果 VM 未运行，应恢复到最近一次已知状态；如果上次状态不可信，应进入 `Error` 并提示用户处理。

补充约束：

- 对于 `Draft`、`Installing`、`Error` 状态的 VM，详情页必须提供显式的“重新选择 ISO”入口。
- 对于已经进入 `Stopped` 或 `Running` 的已安装 VM，默认不展示该入口，避免用户误操作覆盖安装路径。

### 8.7 用户可见的错误恢复动作

MVP 中每类关键错误，至少要映射到一个明确的用户可见动作，而不是只展示报错文案：

- ISO 失效：重新选择 ISO。
- 空间不足：提示释放空间后重试。
- 配置非法：返回配置页面修正。
- 启动失败：重试启动，并允许查看日志。
- 安装中断：重新进入安装流程或删除重建。
- 状态不可信：进入 `Error`，提供重试与删除两条路径。

产品层原则是：错误提示后面必须跟着一个用户能执行的动作。

## 9. 技术边界与实现约束

### 9.1 必须使用 Apple 原生虚拟化技术

必须使用以下官方能力，不引入 QEMU、UTM、Parallels、VMware 作为运行底座：

- `Virtualization.framework`
- `VZVirtualMachineConfiguration`
- `VZGenericPlatformConfiguration`
- `VZEFIBootLoader`
- `VZVirtualMachine`
- `VZVirtualMachineView`

### 9.2 设备模型

MVP 需要的设备能力：

- 存储设备：系统盘 + 安装 ISO。
- 图形设备：用于安装和运行 Linux 桌面。
- 键盘与鼠标设备。
- 网络设备：Virtio + NAT attachment。

MVP 不需要的设备能力：

- 音频设备。
- 共享目录设备。
- SPICE 剪贴板代理。
- Rosetta 目录共享。

### 9.3 entitlement 要求

项目必须配置 Apple 官方要求的 entitlement：

- `com.apple.security.virtualization`

MVP 不需要：

- `com.apple.vm.networking`

原因是 NAT 网络不要求额外网络 entitlement，而 bridged networking 才需要。

### 9.4 架构限制

必须在产品中明确告知用户：

- Apple silicon Mac 只能运行 ARM64 Linux 镜像。
- Intel Mac 只能运行 x86_64 或 AMD64 Linux 镜像。
- 本产品不提供跨架构仿真。

### 9.5 必须预留的扩展点

虽然 MVP 不实现高级能力，但必须在数据模型上为未来版本预留最小扩展空间：

- VM 配置文件要有版本号。
- VM 需要稳定的唯一 ID，而不是仅依赖展示名称。
- 启动源要区分“安装介质启动”和“系统盘启动”。
- 网络模式字段要可扩展，虽然首版只支持 NAT。
- 发行版验证结果要能以配置或文档方式追加，而不是写死在 UI 文案里。

## 10. 非功能要求

### 10.1 稳定性

- 创建后的 VM 可以重复启动，不因应用重启而丢失。
- 安装完成后的 VM 至少应稳定完成：启动、登录、联网、关机。

### 10.2 易用性

- 新建流程不超过一个主向导。
- 默认参数可以让主流 ARM64 Linux 桌面镜像正常安装。
- 错误信息应可读，不暴露纯底层异常给普通用户。

### 10.3 性能

- 首版不追求性能调优极致，但不能出现明显卡死或无响应。
- VM 启动、关闭和创建阶段都应展示明确状态反馈。

### 10.4 数据安全

- 所有 VM 数据默认仅存储于本机。
- 删除操作不可逆，必须显式确认。

### 10.5 可观测性与支持性

为了让 MVP 在内测和小范围使用阶段可诊断、可排错，产品必须具备最低限度的可观测性：

- 所有关键生命周期事件必须写入本地日志，包括创建、启动、停止、失败、删除、状态迁移。
- 日志至少要包含时间戳、VM ID、当前状态、启动源、错误摘要、应用版本、宿主机系统版本。
- 日志默认仅保存在本机，不做自动上传。
- 删除 VM 时，其对应日志也一并清理。

同时必须明确不记录的内容：

- 不记录 guest 屏幕内容。
- 不记录用户键盘输入内容。
- 不记录 guest 内部文件内容。

MVP 至少应提供一种面向用户或测试人员的本地诊断入口，例如打开日志目录或导出诊断信息。

## 11. 验收标准

MVP 完成后，至少满足以下验收条件：

1. 用户可以从应用内创建一台新的 Linux VM。
2. 用户可以选择一个受支持的 ARM64 Linux ISO 并进入图形安装器。
3. 用户可以完成 Linux 安装，并在关机或安装器触发重启后进入已安装系统。
4. 再次启动时，VM 从虚拟磁盘正常引导进入已安装系统。
5. guest 可以通过 NAT 网络访问互联网。
6. 图形界面、键盘、鼠标可正常工作。
7. 应用退出后重新打开，仍能看到并启动之前创建的 VM。
8. 删除 VM 后，本地 bundle 数据被清理。
9. 对镜像不匹配、资源不足、配置非法等情况有明确报错。

### 11.1 验收测试层次

MVP 验收建议分为三层：

- 冒烟验收：创建 VM、进入安装器、完成安装、重启进入系统。
- 回归验收：重复启动/关机、异常退出后恢复、删除后清理。
- 兼容性验收：至少对 1 个主验收镜像和 1 个补充镜像完成验证。

上线门槛：

- 发布阻塞条件：Ubuntu Desktop ARM64 必须完成完整闭环验证。
- 非阻塞补充条件：Fedora Workstation ARM64 最好完成基本可用验证；若未完成，不阻塞 MVP 内测，但对外范围必须只承诺 Ubuntu Desktop ARM64。

### 11.2 MVP 成功指标

除功能验收外，MVP 还需要满足以下产品级成功指标：

- 在主验收环境上，创建、安装、重启进入系统、删除这条主路径必须达到稳定可重复。
- 不能存在已知的数据丢失类问题，例如 VM bundle 异常损坏、删除误删宿主机其他目录。
- 对于 PRD 中定义的关键失败场景，用户必须能通过 UI 完成恢复或明确结束流程，不能依赖手动改文件系统。
- 所有硬性不支持条件都必须在创建前或启动前被明确拦截，而不是在运行中才暴露为不可理解错误。

如果这些指标达不到，产品即使“理论可运行”，也不应被视为 MVP 达标。

## 12. 风险与边界说明

### 12.1 发行版兼容性风险

虽然 Apple 官方支持 Linux VM，但不同发行版在 virtio 驱动、图形体验、安装流程上仍可能存在差异。MVP 不应承诺“所有 Linux 发行版都可用”，而应承诺“对指定镜像完成验证”。

### 12.2 安装流程仍由 guest 控制

Apple 原生虚拟化能负责启动和运行安装器，但 Linux 发行版的安装步骤由 guest 自身决定。MVP 不负责改造发行版安装器，也不承诺无人值守安装。

### 12.3 后续扩展会明显增加复杂度

以下功能一旦进入范围，复杂度会明显上升：

- 多 VM 管理
- 快照
- Bridged networking
- 共享目录
- 剪贴板同步
- Rosetta
- Intel 兼容矩阵

因此应严格守住 MVP 边界。

### 12.4 宿主机生命周期风险

除 guest 自身问题外，MVP 还要承认宿主机生命周期会影响 VM 体验：

- 宿主机睡眠、重启、异常关机，可能导致 VM 处于不可信状态。
- 应用升级或宿主系统升级后，历史 VM bundle 仍必须被保留和识别。
- 当产品无法可靠判断 VM 当前一致性时，应优先进入 `Error` 并要求用户显式重试，而不是假设状态正确。

首版不追求复杂的自动修复机制，但必须优先保证数据不被误删、状态不被误判。

为了让后续设计和开发更可控，建议把 MVP 拆成以下里程碑：

- 锁定 bundle 结构和状态模型。

### Milestone 2: 单 VM 创建与安装闭环

### Milestone 3: 运行期能力与错误恢复

- 可以稳定启动、关闭、强制停止、删除。

- 可以在应用重启后恢复 VM 元数据。

### Milestone 4: 发布前验证

- 完成主验收镜像的全流程验证。
- 完成至少一个补充镜像验证。

## 15. 关键决策锁定

以下 5 项此前的未决问题，现统一锁定为 MVP 决策：

### 15.1 首版验收发行版

- 首版唯一对外承诺的主验收发行版锁定为 Ubuntu Desktop ARM64（当前受支持 LTS）。
- Fedora Workstation ARM64 保留为补充验证镜像，但不是发布阻塞项。

原因：

- 只锁定一个主发行版，可以显著降低安装流程、图形驱动、virtio 兼容性带来的测试矩阵复杂度。
- Ubuntu Desktop ARM64 的用户认知和桌面安装路径更适合作为首版 MVP 的标准闭环。

### 15.2 VM bundle 默认存储路径

- 首版固定使用 `~/Library/Application Support/phas/VMs/<vm-id>.vmbundle/`。
- MVP 不支持用户自定义 bundle 存储路径。

原因：

- 固定路径有利于实现、排错、清理和文档说明。
- 自定义路径会引入文件权限、移动磁盘、路径失效和数据管理复杂度，不符合 MVP 收敛原则。

### 15.3 是否提供“重新选择 ISO”入口

- 提供。
- 该入口仅对 `Draft`、`Installing`、`Error` 状态可见。
- 已安装 VM 默认不展示该入口。

原因：

- 这是处理 ISO 丢失、安装中断和镜像选择错误的最低成本恢复手段。
- 同时通过状态限制，避免进入已安装 VM 的误操作路径。

### 15.4 是否展示宿主机资源信息

- 展示，但只展示轻量级决策信息。
- MVP 只展示总内存、推荐预设、不可选项原因等与创建决策直接相关的信息。
- 不做复杂实时监控或系统诊断视图。

原因：

- 完全不展示宿主机信息，会让用户在资源选择上缺少依据。
- 展示过多系统细节又会稀释 MVP 的产品重点。

### 15.5 首发分发方式

- 首版按“开发者签名后的直接分发”设计，可支持内部测试和小范围试用。
- 不以 Mac App Store 首发为目标，也不让 Mac App Store 约束反向影响 MVP 设计。

原因：

- 直接分发更适合快速验证 Virtualization.framework 产品闭环。
- 这样可以减少首版在审核、分发策略和商店约束上的不确定性，把精力集中在 VM 体验本身。

结论：

以上 5 项已经从“未决问题”转为“已锁定决策”，后续技术设计应直接以此为输入，不再重复讨论同一级别的产品取舍。

## 16. 研究依据

以下结论主要来自 Apple 官方文档：

- Virtualization.framework 总览
- Running Linux in a Virtual Machine
- Creating and Running a Linux Virtual Machine
- Running GUI Linux in a Virtual Machine on a Mac
- VZLinuxBootLoader
- VZNATNetworkDeviceAttachment
- VZBridgedNetworkDeviceAttachment
- Shared directories
- Adding the Virtualization Entitlement to Your Project
- Running Intel Binaries in Linux VMs with Rosetta

### 16.1 官方一致性结论

基于对 Apple 官方 Virtualization 文档的逐条复核，本 PRD 当前结论可分为三类：

#### A. 与 Apple 官方定义直接一致的内容

- Apple 官方明确支持在 Apple silicon 和 Intel Mac 上运行 Linux VM，但 guest 镜像架构必须与宿主机架构匹配。
- Apple 官方同时提供两条 Linux 路径：
- 路径一：基于 `VZLinuxBootLoader` 的 kernel + initrd 路径。
- 路径二：基于 ISO 安装镜像、`VZEFIBootLoader`、图形设备和 `VZVirtualMachineView` 的 GUI Linux 路径。
- `VZNATNetworkDeviceAttachment` 不要求 `com.apple.vm.networking` entitlement。
- `VZBridgedNetworkDeviceAttachment` 需要 `com.apple.vm.networking` entitlement，否则配置无效。
- Linux 共享目录依赖 VirtioFS 能力。
- 剪贴板共享基于 SPICE agent 能力，Linux guest 侧需要 `spice-vdagent`。
- Rosetta for Linux 仅适用于 Apple silicon 上的 ARM Linux VM，用来运行 x86_64 Linux 二进制，不用于安装 Intel Linux 发行版。
- 使用 Virtualization.framework 必须启用 `com.apple.security.virtualization` entitlement。

以上内容属于 Apple 官方能力边界，本 PRD 与官方文档一致。

#### B. PRD 的产品收敛决策，不是 Apple 官方硬要求

- 首版仅支持 Apple silicon。
- 首版仅锁定 Ubuntu Desktop ARM64 为主验收发行版。
- 首版仅做单 VM 闭环。
- 首版固定 VM bundle 存储路径。
- 首版选择 macOS 14+ 作为产品支持基线。
- 首版不暴露高级网络、共享目录、剪贴板、Rosetta 等扩展能力。

这些内容不是 Apple 官方 API 的强制限制，而是为了缩小实现范围、测试矩阵和支持成本而做出的产品决策。它们在方向上与官方技术体系并不冲突，且符合 MVP 收敛原则。

#### C. 经过本轮复核后修正的表述

- Apple 官方 GUI Linux 示例路径的最低系统前提是 macOS 13.0+；本 PRD 保留 macOS 14+，但现已明确这是产品支持收敛，而非官方 API 最低要求。
- Linux 安装完成后的结束形态不应只写成“用户关机”；安装器也可能直接重启并从系统盘进入已安装系统。因此本 PRD 已补充 `Installing -> Running` 路径，并相应修正首次安装流程与验收文案。

#### D. 结论

本 PRD 的核心实现路线与 Apple 官方 Virtualization 技术方向一致，没有出现与官方设备模型、启动路径、entitlement 规则或平台边界相冲突的设计。

当前 PRD 中需要特别注意的地方，不是“技术路线错误”，而是以下产品层现实：

- Apple 提供的是框架能力，不提供对所有 Linux 发行版的产品级兼容承诺。
- GUI 安装路径可行，但不同发行版的安装器行为、重启流程、驱动体验仍需靠产品自己的验证矩阵确认。
- 因此，本 PRD 采取“官方能力之上做产品收敛”的策略是合理的。

### 17.4 推荐实现顺序

为了和 MVP 范围保持一致，后续技术设计建议按以下顺序落模块：

1. A + B + C + D：先打通“创建一台 VM”的前半段。
2. E + F + G：再打通首次安装、状态转移和图形交互。
3. H + K：补全 NAT 联网和详情页动作入口。
4. I + J：最后补齐错误恢复和本地诊断能力。

这个顺序的核心目的是：优先形成单 VM 的创建、安装、启动闭环，再补恢复和支持能力，而不是一开始就分散到高级功能上。

本 PRD 的核心产品判断是：

Apple 原生虚拟化完全可以支撑一个本地 Linux VM 产品，但首版必须克制范围。最合适的 MVP 不是“只证明能跑”，而是“只做一台 VM，但把创建、安装、启动、持久化、联网这条链路做完整”。
