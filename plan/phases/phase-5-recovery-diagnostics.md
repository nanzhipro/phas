# Phase 5: Add Recovery Flows, Diagnostics, and App Relaunch Restoration

## 阶段定位

在 phase-4 已经具备单 VM 运行窗口、详情页和生命周期控制之后，本阶段负责补上产品闭环里最容易让用户失去信心的部分：应用重启后的状态恢复、脏状态显式报错、最小诊断面板和明确的恢复动作。

## 必带上下文

- plan/common.md
- Phase 4 已完成
- PRD.md 中关于错误显式恢复、诊断可见性、应用 relaunch 后状态一致性和单 VM 数据安全的要求

## 阶段目标

- 交付应用 relaunch 后的单 VM 状态恢复逻辑，确保首页和运行窗口入口能基于 bundle 重新建立产品态。
- 当持久化状态处于 `Running` / `Installing` 等瞬时态但缺少真实运行会话时，明确映射到 `Error`，而不是伪造“仍在运行”。
- 提供最小诊断面板，展示当前恢复结论、最近错误摘要、bundle/log 路径和可执行恢复动作。
- 提供显式恢复动作，例如重新装配运行会话、把脏状态回收为安全可重试状态，或引导用户查看日志后重试。

## 实施范围

- 应用启动后的 VM 记录恢复与运行窗口入口恢复。
- 脏状态识别、恢复决策模型和用户可见的恢复动作。
- 首页或详情页中的诊断/恢复面板。
- 与上述行为直接对应的测试与说明文档。

## 本阶段产出

- 一套 relaunch-aware 的 VM 恢复逻辑。
- 一套把不可信的瞬时态映射到 `Error` 的恢复规则。
- 可见的诊断信息与恢复动作，不再只靠通用 alert 文案。
- 覆盖恢复映射、诊断快照和恢复动作策略的自动化测试。

## 明确不做

- 不实现 phase-6 的验证矩阵自动化或 phase-7 的 release 文档/接受台账。
- 不实现多 VM 恢复编排、批量运维、日志归档上传或远程诊断。
- 不把恢复实现成静默自动修复，更不能偷偷删除 bundle 或重置数据。

## 完成判定

- 应用重启后，已持久化的 VM 记录会重新出现在首页与详情视图中，运行窗口入口能够恢复到正确可用/不可用状态。
- 对于无活跃会话却落在 `Running` / `Installing` 等瞬时态的记录，产品会进入 `Error` 并给出明确恢复动作，而不是继续显示为正常运行。
- 诊断面板能展示恢复结论、最近错误、bundle/log 路径和当前可执行的恢复动作。
- 自动化测试覆盖 relaunch 状态映射、恢复动作规则或诊断快照；`xcodebuild test` 通过。
